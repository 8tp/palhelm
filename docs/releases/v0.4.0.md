# Palhelm v0.4.0 — Integration API

v0.4.0 adds a read-only, bearer-token API for Discord bots, dashboards, and scripts, alongside
the existing session-authenticated panel API. It does not deploy, restart, or modify the live
Palworld service during development.

## What's new

A dedicated, GET-only surface at `/api/integration/v1` — `players`, `players/{uid}`, `pals`,
`guilds`, `map`, `server`, `metrics/current` — authenticated by a bearer key created and revoked
from **Settings → Integration API** (admin only). It is a separate namespace, not dual-auth on
the existing session API: the bearer group carries no reference to the session cookie or
`adminOnly`, and no session-gated handler can ever observe a bearer request as authenticated. The
reverse holds too — a bearer key opens nothing under `/api/v1/*`. A full-router enumeration test
proves both directions against the real router, not a hand-kept list.

## Token semantics and storage

- Keys look like `phk_<8 lowercase hex>_<43 base64url chars>` — 56 characters total, `phk_`
  grep-able in leaked configs and secret-scanning tools. The 8-hex-char `id` prefix is the public,
  non-secret handle used in logs, audit events, and the Settings UI.
- The server stores only `SHA-256(full plaintext key)`. The plaintext is returned **exactly
  once**, in the create response, and is never persisted, logged, or audited again. Validation is
  in-memory and constant-time: a `subtle.ConstantTimeCompare` against the stored digest, or an
  equal-cost dummy compare when the key id is unknown, so a found-vs-not-found timing signal
  cannot leak which ids exist.
- Every auth failure mode — missing header, malformed header, unknown id, wrong secret, revoked
  key — returns the byte-identical `401` with `WWW-Authenticate: Bearer`. No response variant can
  be used to probe key validity.
- Revocation is soft (`DELETE /api/v1/integration-keys/{id}`): the row is retained forever with
  `revokedAt` set, for the audit trail, and the in-memory cache flips synchronously in the same
  request — the very next bearer request for that key fails, no restart and no staleness window.
  There is no hard delete and no un-revoke; re-issuing means creating a new key. Active keys are
  capped at 100; creating past the cap is `409 too_many_keys`.

## Redaction policy

Assume every response ends up pasted in a public channel: the surface is strictly a subset of
what the authenticated viewer sees, never more, and redacted fields are **absent**, never `null`.
Never exposed on this surface, in any endpoint, under any name: `steamId` and other platform ids,
`accountName`, `ping`, live player `location`, `banned`, `whitelisted`, per-player `sessions`,
`worldGuid`, and `panelVersion`. Base locations under `/guilds` *are* exposed — persistent,
communal structures already discoverable in-game, unlike a live player position. Enforcement is
structural: every integration endpoint serializes a dedicated allowlist view struct; there is no
code path from the underlying store/poller types into a token response.

## Rate limits and abuse controls

60 requests/minute per key by default, a sliding window, overridable via
`PALHELM_INTEGRATION_RATE_LIMIT`. The limiter is keyed on the key id, not the network address, so
spoofed `X-Forwarded-For`/`X-Real-IP` have no effect. No state of any kind — no limiter bucket, no
cache mutation — is allocated before a token passes constant-time validation, so rotating garbage
tokens cannot grow any map. Limiter cardinality tracks the active-key count (≤ 100) even under
revoke/create churn, because revocation deletes the key's bucket in the same step as the cache
flip; a bounded 1024-entry map remains as defense in depth and fails closed (`429`) if it is ever
full. Every `429` carries `Retry-After`.

## Pagination and freshness

`/players` and `/pals` are keyset-paginated (`uid` / `instance_id`, both immutable save-derived
GUIDs) rather than offset-paginated, so a save re-parse mid-walk cannot produce duplicates or
gaps. `nextCursor` is always the key of the last row *returned*, and is `null` whenever a page
returned fewer than `limit` rows — an empty `data` array is never paired with a non-null cursor.
Each page is internally consistent (a re-parse swaps tables in one transaction); across pages, no
row present for the whole walk is ever duplicated or skipped, though rows created or deleted
mid-walk may or may not appear — the standard keyset contract, and the per-response
`lastParseAt` lets a client detect a re-parse landing mid-walk. Every `200` carries a weak
content-hash `ETag`; sending it back as `If-None-Match` yields a bodyless `304`, cutting bandwidth
for bots that poll on a timer (the server still runs the query — a `304` saves bandwidth, not
work). `Cache-Control: no-store` applies unconditionally to every response on this surface, and
now also to every `/api/v1/integration-keys` response, since its `201` is the only response in
the whole panel that ever carries a plaintext credential.

## No CORS, TLS guidance

There is no CORS support anywhere in the panel — no `Access-Control-Allow-*` header is ever set.
A browser-based dashboard on another origin must proxy Integration API calls server-side rather
than call the API directly from JavaScript, keeping the bearer key out of browser-held code. As
with the rest of the panel, **serve this API over TLS at the network edge and treat these keys
like passwords** — a bearer key sent over cleartext HTTP is readable by anyone on the path, and
the panel's trusted-network-edge posture is the operating assumption, not a substitute for TLS
termination.

## Operator key-creation flow

Log in as admin, open **Settings → Integration API**, and create a key with a label (for example
`discord-bot`). The plaintext key is displayed once, in a copy field with an explicit "shown
once" warning, and is discarded from the browser's own state on dismiss. Revoke a key from the
same card at any time. See [docs/API.md](../API.md#integration-api-bearer-tokens) for the full
endpoint reference and a curl example.

## Database migration and rollback

v0.4.0 introduces the panel's first schema migration since `001_init.sql`: `002_api_keys.sql`
adds the `api_keys` table backing the feature above. A small embedded migration runner now backs
`store.Open()` (numbered `migrations/NNN_name.sql` files, version tracked in the existing
`kv.schema_version` row, each migration applied in its own `BEGIN IMMEDIATE` transaction with an
in-transaction version re-read so a concurrently-applied migration is a no-op rather than a double
application). A fresh database runs every migration; an existing v0.3.0 database runs only the
new one and keeps its player/console/event history intact — the upgrade test verifies this against
an exact v0.3.0 schema, not an approximation.

**The runner fails closed on a database newer than the binary knows about**: if `kv.schema_version`
exceeds the newest embedded migration, `Open()` refuses to start rather than silently opening or
partially migrating a future-versioned database. **Rollback to the v0.3.0 binary is supported and
requires no manual step:** the v0.3.0 binary's `Open()` runs only `001_init.sql`, which is
entirely `IF NOT EXISTS` / `INSERT OR IGNORE`, so it opens a v0.4.0 database cleanly.
`schema_version` stays `2`, and the `api_keys` table simply sits unused — any keys created under
v0.4.0 remain dormant and work again immediately on re-upgrade, which resumes at version 2 with no
migration re-run. Back up the data directory before any release change, as always.

## Known limitations

- The in-memory key-validation cache and the per-key rate limiter are process-local by design;
  multi-replica deployments need shared state, same caveat as the v0.3.0 login limiter.
- No CORS support: browser dashboards must proxy Integration API calls server-side.
- No webhooks (Discord join/leave push), no Server-Sent Events for bearer tokens, and no write
  scopes or scoped keys — every key can read every endpoint in the group, there is no narrower
  grant. These are deferred future work, not overlooked.
- Guild listing is not paginated; guild counts are expected to stay small (tens), and pagination
  there is a candidate for a future release if a world proves otherwise.
- No per-key rate-limit overrides and no key expiry dates; every key uses the same
  `PALHELM_INTEGRATION_RATE_LIMIT` ceiling until explicitly revoked.
