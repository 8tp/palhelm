# Palhelm v0.3.0 — Safe Operations

v0.3.0 is a safety and contract-correctness release. It closes the post-v0.2.0 release blockers
without deploying, restarting, or modifying the live Palworld service during development.

## Security

- Viewer sessions can no longer retrieve raw Config or browser-visible game credentials. Both
  game passwords are write-only placeholders, and sensitive responses use `Cache-Control:
  no-store`.
- Login throttling uses the transport peer unless an explicit trusted-proxy CIDR permits
  forwarding headers. Limiter state expires and has a global cardinality bound.
- The panel now emits CSP, content-type, frame, referrer, and permissions hardening headers.
  Map tiles require a session and use private caching.
- `PALHELM_SECURE_COOKIES=true` explicitly forces Secure cookies behind TLS termination. Trusted
  proxies may also report HTTPS through `X-Forwarded-Proto`.

## Compose mount migration

The v0.2 single-file mount is unsupported for Config writes. Atomic replacement requires sibling
temporary and backup files, which a file bind cannot provide safely.

1. Create a narrow host directory such as `./compose`, owned by the UID/GID used by Palhelm.
2. Move the project file to `./compose/docker-compose.yml`.
3. Adjust relative bind sources because Compose now resolves them from `./compose` (the shipped
   example uses `../data` and `../palhelm-data`).
4. Mount the directory read-write as `./compose:/compose` and set
   `PALHELM_COMPOSE_FILE=/compose/docker-compose.yml`.
5. Run Compose commands against the moved file (its directory is the project context), for
   example `docker compose -f ./compose/docker-compose.yml up -d palworld` from the parent
   directory, or `docker compose up -d palworld` from inside `./compose`.

Startup probes single-file mounts, file availability, directory writability, and sibling rename
support. The API/UI remains explicitly read-only with a reason when the capability is unavailable.

## Config and Docker apply

Backend, TypeScript, and mock contracts now share typed values, key/group names,
`editable`/`readOnly`, full GET/PUT documents, version conflicts, and one error envelope.
Concurrent updates are serialized; external file changes are detected with a SHA-256 content
version. Changed strings are safely YAML-encoded, control/newline input is rejected, and the full
result is parsed and semantically checked before atomic rename.

One-click Docker apply is intentionally disabled in v0.3.0. Palhelm cannot generally preserve the
host Compose project identity or relative host-path semantics from inside its container. The UI
returns the manual host command instead; no Docker socket mount is needed or recommended.

## Backups and saves

- Backup, dry-run, restore, and pre-restore safety copies resolve the active world from the
  normalized REST world GUID and fail closed on mismatch or ambiguity. Only REST transport
  unavailability allows newest-directory fallback, which is marked in the archive and event.
- Conflicting create/delete/prune/dry-run/restore work is excluded. Walk, copy, hash, extraction,
  and shutdown paths observe cancellation.
- Archive extraction limits entry count, individual entry size, total expanded bytes, path shape,
  and actual copied bytes.
- Backup schedules persist their actual next deadline, reset immediately on PUT, and disarm
  immediately when disabled.
- SAV parsing now applies global decoded-node and allocation budgets, absolute collection/blob
  limits, typed primitive arrays, drift reporting for unexpected skips, and fuzz coverage.

## Whitelist behavior

The existing `/whitelist` storage is not authoritative game-server access control. The UI now
labels it as local player notes and explicitly states that it does not determine who may join.
Authoritative enforcement remains deferred until a supported vanilla or verified PalDefender
integration is available.

## Shutdown and restart semantics

The existing action is now named graceful shutdown. It can stage announcements and be cancelled
before the final REST shutdown call. Palhelm cannot start the stopped server or observe a restart,
so it does not claim scheduled restart support. A separately configured and verified supervisor or
container restart policy may restart Palworld, but that behavior is external to Palhelm.

## Database migration and rollback

v0.3.0 makes no SQLite schema change; existing v0.2.0 databases open in place, and the upgrade test
verifies stored player/KV data remains readable. No database rollback step is required. Back up the
data directory before any release change as normal. Config writes retain one daily
`.palhelm.bak` sibling; rolling back the binary does not undo host Compose edits.

## Other corrections

- Background pollers/backups join before SQLite closes, and SSE streams close at application
  shutdown.
- Config and Backups OpenAPI methods, responses, HEAD behavior, and schemas are documented and
  contract-tested.
- THGL zoom uses the active layer tile size; absent map layers normalize to `[]`; dialogs have
  accessible names; browser 401 handling clears stale auth state and avoids inappropriate retries.
- The GET-only `game-data` re-probe still returned 404 on Palworld `v1.0.0.100427`; no client work
  was started.

## Deferred and known limitations

- Verified restart service, Prometheus exporter, `building_count`, and toast improvements remain
  deferred. Discord, Paldeck regeneration/screen, Combobox, teleport, and `game-data` consumption
  remain out of scope.
- Login limiter state is process-local; multi-replica deployments need shared rate state.
- An uncooperative external editor can still write in the narrow interval between the final
  compose content check and atomic rename.
- Offline restore uses the last REST-observed poller GUID before the warned mtime fallback. After
  an out-of-band world switch, let Palhelm observe the new world and run the dry-run before
  stopping the game server.
- The 32 GiB aggregate extraction ceiling is a hard upper bound, not a free-space guarantee;
  operators with smaller data volumes must verify capacity before restoring an imported archive.
- Map/player fixture coordinates remain illustrative until a verified 1.0 dataset is available.
