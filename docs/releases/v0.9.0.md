# Palhelm v0.9.0

Palhelm 0.9.0 turns the save's authoritative per-player Pal capture records into a
viewer-safe Paldeck experience, adds connected guild exploration, and makes dense
live maps easier to navigate without weakening coordinate accuracy or privacy. It
also finishes a plain-English copy pass across every panel page and replaces the last
placeholder operational numbers with real values.

## Paldeck and guild exploration

- Save-observed capture counts and Paldeck unlock flags are retained per player.
- Server and player progression explicitly report observation coverage, unavailable
  data, parser bounds, and catalog drift instead of inferring catches from owned Pals.
- Boss identifiers are folded into their base species for progression while captured
  specimens keep their Boss presentation in the Pal explorer.
- Dedicated guild views connect members, bases, Pals, activity, and map focus links.
- Pal explorer filters live in canonical, shareable URLs so records, guilds, and other
  panel pages can open an exact roster view without sharing pagination cursors.

## Base camps, names, and locations

- Base world locations and the player's chosen base name are decoded from
  `BaseCampSaveData.RawData` (`backend/internal/sav/world.go`).
- An unnamed base — the engine writes an untranslated placeholder template into every
  base the player never renamed — normalizes to `null`, never `""` or a synthetic
  label. The panel shows a `Base N` fallback for those rows.
- A base whose world transform could not be decoded serves a `null` location, never a
  misleading `(0, 0)`.
- Map base markers label with the base's own name, falling back to the guild display
  label (including the unnamed-guild member fallback).

## Pal condenser star ratings

- Each Pal now carries a nullable condenser `rank` (1–5; displayed stars are `rank − 1`)
  decoded from the save, exposed on player detail Pals, the Pal explorer, guild detail
  Pals, and the Integration API.
- A missing rank renders as `Unavailable` or nothing — never as zero stars. Soul-
  enhancement stats (`Rank_HP`/`Rank_Attack`/`Rank_Defence`) are out of scope.

## Guild list filter

- The guild list shows only guilds with at least one placed base **and** at least one
  confirmed player. Membership evidence counts from either direction: a group-roster
  member that resolves to a known player, **or** a known player whose own save record
  points back at the guild (`guildListRealFilter` in
  `backend/internal/store/store.go`). Real 1.0 saves carry base-owning guilds with an
  empty group roster whose players still reference them, so the back-reference matters.
- The filter drops Palworld's non-guild group records (a solo player's auto-created
  organization and other group types) so every consumer — guilds page, players "Guilds"
  tab, dashboard count, map bases — agrees on the same set.
- Guild detail deliberately does **not** apply the filter, so a player row can still open
  its guild even when the guild is filtered out of the list.

## Map and build reliability

- The layer bounds check compared world axes in swapped order; the fix places and
  filters markers correctly on Palpagos versus the World Tree (a World Tree visitor now
  renders on the tree layer), and Feybreak positions render correctly.
- Same-layer player and base markers cluster independently at dense screen positions;
  exact markers remain reachable and selected markers are never swallowed by a cluster.
- The players page "Show on map" action deep-links to `/map?x=&y=`, and the player
  detail Position field is labeled as last-save data.
- Unnamed guilds display as "<member>'s guild" (`frontend/src/app/guildDisplay.ts`).
- Version-pinned transform, inverse-coordinate, bounds, and layer-boundary fixtures
  protect Palpagos and World Tree alignment in CI.

## Honest operational data

- `GET /api/v1/backups/storage` reports the backup filesystem's real capacity and free
  space via `statfs(2)`, and reports `null` on failure — the fabricated "of 50 GB"
  denominator is gone. The backups and diagnostics screens read this and say when a
  build cannot report it rather than inventing a figure.
- `GET /api/v1/server` now includes `sessionDays` and `saveSyncMinutes`, and
  `PALHELM_SESSION_DAYS` is now actually honored (login sessions were previously
  hardcoded to 7 days regardless of the variable).
- Real `docs.palhelm.com` and `github.com/8tp/palhelm` links replace the placeholder
  hrefs in the settings About card.

## Copy and layout

- A plain-English copy and layout pass ran across every panel page. Some previously
  quoted UI strings changed; for example the Paldeck heading is now
  "Paldeck · capture progress from parsed saves" rather than "Save-observed Paldeck".

## Developer tooling and CI

- A committed Playwright smoke harness (`npm run test:smoke`, `frontend/tests/smoke/smoke.mjs`)
  drives the routes derived from `NAV_ITEMS` and asserts no console errors and no
  horizontal overflow, backed by a CI smoke job. SSE is disabled in mock mode.

## API additions

- `GET /api/v1/server`: adds `sessionDays` and `saveSyncMinutes`.
- `GET /api/v1/backups/storage`: `{totalBytes, freeBytes}`, each nullable.
- `GET /api/v1/guilds/{id}`: viewer-safe guild detail (members, bases with nullable name
  and location, associated Pals with condenser rank, bounded 30-day panel-observed
  activity).
- `GET /api/v1/paldeck` and `GET /api/v1/players/{uid}/paldeck`: viewer-safe capture
  progression against the pinned Palworld 1.0 catalog.
- Pal objects across the session and Integration APIs gain a nullable `rank`; guild
  bases gain a nullable `name`.

## Upgrade and rollback

This release advances the SQLite schema from version 10 to 12: migration `010`
(per-player Paldeck observations), `011` (nullable base-camp `name`), and `012` (nullable
Pal `rank`). Back up the complete Palhelm data volume before upgrading.

The migration runner fails closed: an older binary refuses to open a database written at
a newer schema. Rolling the image back to 0.8.x therefore requires restoring the
pre-upgrade data-volume backup — changing only the image tag is not sufficient, because
the old binary will not start against a schema-12 database. Palhelm never runs schema
migrations backward. See [Updating Palhelm](https://docs.palhelm.com/getting-started/updating/)
for the full procedure.

This release changes only Palhelm and its authenticated API. It does not require or
perform a Palworld game-server restart.
